# План разработки сервиса согласования контента (approval-service)

## КРИТИЧЕСКОЕ СИСТЕМНОЕ ОГРАНИЧЕНИЕ (ОБЯЗАТЕЛЬНО К ИСПОЛНЕНИЮ)

* **Интеграция с MCP-сервером google-stitch:** Весь UI/UX дизайн, прототипы интерфейсов и связанные с ними структуры данных (если они влияют на визуальное представление или логику переходов на фронтенде) должны запрашиваться **исключительно** через инструменты (tools) MCP-сервера `google-stitch`.
* **Запрет на автономный дизайн:** Категорически запрещается проектировать элементы интерфейса, схемы экранов или логику взаимодействия «от себя». Разрешено использовать только готовые прототипы со ститча.
* **Синхронность взаимодействия:** При вызове тулов MCP-сервера исполнитель (или AI-агент) **ОБЯЗАН** дожидаться полного ответа от `google-stitch`. Переходить к реализации без подтвержденного ответа от MCP-сервера запрещено.

## Фаза 1: Инициализация проекта, базовая инфраструктура и системные эндпоинты

**Цель:** Создать фундамент проекта, настроить управление зависимостями, контейнеризацию и базовый роутинг для проверок состояния.
**Контекст для исполнителя:** Требуется поднять пустой backend-сервис на Python + FastAPI. База данных пока не нужна. Сервис должен уметь отвечать на health-чеки. Любые базовые заготовки под будущую интеграцию интерфейсов сверять с прототипом из `google-stitch`.

**Задачи:**

1. Инициализировать проект на Python (рекомендуется использовать `Poetry` или `pip` + `requirements.txt`).
2. Настроить базовый `FastAPI` application.
3. Реализовать системные эндпоинты:
* `GET /health` (возвращает статус 200 OK).
* `GET /ready` (возвращает статус 200 OK).


4. Настроить Docker:
* Создать `Dockerfile` для локального запуска сервиса.
* Создать `docker-compose.yml` (пока только с одним сервисом `app`).


5. Настроить линтеры и форматтеры (flake8, black/ruff).

**Git-флоу:**

* `git init` (если репозиторий новый).
* `git checkout -b feature/init-project`
* Добавить `.gitignore` для Python-проектов.
* Сделать коммит: `git commit -m "chore: init project, add docker and health endpoints"`
* `git push` и мердж в `main`.

## Фаза 2: Проектирование БД и системы миграций

**Цель:** Настроить подключение к базе данных и создать структуру таблиц для заявок и логов аудита.
**Контекст для исполнителя:** Сервис работает в мульти-тенантной среде (workspaces). Необходимо спроектировать таблицы для хранения "Заявок на согласование" (Approval Requests) и "Истории изменений" (Audit Trail). База для локальной разработки — SQLite или PostgreSQL. Используется ORM (например, SQLAlchemy) и система миграций (Alembic).

**Задачи:**

1. Добавить сервис БД (`postgres` или настроить локальный файл `sqlite`) в `docker-compose.yml`.
2. Инициализировать Alembic для миграций (`alembic init`).
3. Описать модели данных (убедиться, что типы полей соответствуют контрактам данных, полученным из прототипов через MCP-сервер `google-stitch`):
* **Таблица `approval_requests**`:
* `id` (UUID/String, PK)
* `workspace_id` (String, индекс для изоляции тенантов)
* `source_type` (String: publication, scenario, edit, external)
* `source_id` (String)
* `title` (String)
* `description` (String, nullable)
* `reviewers` (JSON/Array of strings — ID пользователей)
* `status` (String: PENDING, APPROVED, REJECTED, CANCELED)
* `idempotency_key` (String, уникальный ключ для предотвращения дублей)
* `created_at`, `updated_at` (Timestamps)


* **Таблица `audit_logs**`:
* `id` (PK), `request_id` (FK), `actor_user_id` (String), `action` (String), `payload` (JSON), `created_at` (Timestamp).




4. Сгенерировать и применить первую миграцию.

**Git-флоу:**

* `git checkout main` -> `git pull`
* `git checkout -b feature/database-schema`
* Коммит: `git commit -m "feat: setup alembic and add initial db schema for requests and audit"`
* Мердж in `main`.

## Фаза 3: Авторизация (Stub) и Middleware изоляции

**Цель:** Реализовать заглушку для проверки прав доступа и изоляции данных по рабочим пространствам.
**Контекст для исполнителя:** Сервис не реализует полноценный OAuth. Вместо этого ожидается авторизационная заглушка (например, парсинг кастомного заголовка `X-Auth-Mock`), в которой передается информация о текущем пользователе, его `workspace_id` и списке прав.

**Задачи:**

1. Создать FastAPI Dependency (middleware/функцию), которая читает данные авторизации из запроса.
2. Формат ожидаемых входных данных (выбрать и задокументировать один, например, JSON в заголовке):
`{"user_id": "usr_123", "workspace_id": "ws_1", "permissions": ["approval:read", "approval:create"]}`.
3. Реализовать функции-валидаторы, которые будут падать с `403 Forbidden`, если:
* Запрошен чужой `workspace_id` (не совпадающий с токеном).
* Отсутствует требуемое право (например, нет `approval:create` при попытке создать заявку).


4. Настроить маскирование логов (исключить попадание секретов/токенов в stdout).

**Git-флоу:**

* `git checkout -b feature/auth-stub`
* Коммит: `git commit -m "feat: add auth stub middleware and permission validators"`
* Мердж в `main`.

## Фаза 4: Базовый бизнес-API (Создание и чтение заявок)

**Цель:** Реализовать CRUD-операции для заявок с учетом идемпотентности и прав доступа.
**Контекст для исполнителя:** Нужны эндпоинты для создания и чтения заявок. Схемы ответов и структура полей в JSON должны строго соответствовать логике экранов, полученной из ответов MCP-сервера `google-stitch`. Дождаться ответа от Ститча перед фиксацией схем Pydantic.

**Задачи:**

1. Реализовать `POST /api/v1/workspaces/{workspace_id}/approval-requests`:
* Требует право: `approval:create`.
* Body: `sourceType` (enum), `sourceId`, `title`, `description`, `reviewerUserIds`.
* Идемпотентность: принимать заголовок `Idempotency-Key`. Если ключ уже существует — возвращать ранее созданную сущность (200 OK), иначе создавать новую со статусом `PENDING` (201 Created).
* Писать событие создания в таблицу `audit_logs`.


2. Реализовать `GET /api/v1/workspaces/{workspace_id}/approval-requests`:
* Требует право: `approval:read`.
* Возвращает список заявок строго для указанного `workspace_id`.


3. Реализовать `GET /api/v1/workspaces/{workspace_id}/approval-requests/{request_id}`:
* Требует право: `approval:read`.
* Возвращает конкретную заявку, либо 404.



**Git-флоу:**

* `git checkout -b feature/crud-requests`
* Коммит: `git commit -m "feat: implement create and read endpoints with idempotency"`
* Мердж в `main`.

## Фаза 5: Машина состояний (Принятие решений)

**Цель:** Реализовать логику переходов статусов заявки и жесткие ограничения машины состояний.
**Контекст для исполнителя:** Заявка может быть согласована, отклонена или отменена. Поля форм (комментарии, причины отказа) синхронизированы с требованиями интерфейсов прототипа `google-stitch`. Каждое действие требует записи в аудит-лог.

**Задачи:**

1. Реализовать `POST .../{request_id}/approve`:
* Право: `approval:decide`.
* Body: `{"comment": "string"}` (обязательно).
* Логика: Если статус уже финальный (не PENDING) -> ошибка 400 (или 422). Иначе смена на `APPROVED`.


2. Реализовать `POST .../{request_id}/reject`:
* Право: `approval:decide`.
* Body: `{"reason": "string"}` (обязательно).
* Логика: Проверка статуса, смена на `REJECTED`.


3. Реализовать `POST .../{request_id}/cancel`:
* Право: `approval:cancel`.
* Body: `{"reason": "string"}` (обязательно).
* Логика: Проверка статуса, смена на `CANCELED`.


4. Во всех 3-х эндпоинтах добавлять запись в `audit_logs` с указанием, какой `user_id` принял решение и с какой причиной/комментарием.

**Git-флоу:**

* `git checkout -b feature/state-machine`
* Коммит: `git commit -m "feat: implement approve, reject, cancel workflows and strict state transitions"`
* Мердж в `main`.
