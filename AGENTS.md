# Инструкции и роли агентов (Agents Pipeline)

## ГЛОБАЛЬНЫЕ ПРАВИЛА И ОГРАНИЧЕНИЯ ДЛЯ ВСЕХ АГЕНТОВ

* **Системный приоритет (Opencode):** Вся разработка ведется с нуля. Главная архитектурная и функциональная цель проекта — создание «новой» кнопки (плагина) для терминального интерфейса Opencode. Инфраструктура, пайплайны и эндпоинты из `PLAN.md` и `SPEC.md` применяются строго для обслуживания этой задачи.
* **UI/UX Дизайн (Критично):** Категорически запрещено проектировать дизайн, элементы интерфейса или логику переходов на фронтенде автономно. Любые визуальные структуры и контракты данных запрашиваются **исключительно** через MCP-сервер `google-stitch`. Агент обязан дождаться полного синхронного ответа от сервера перед написанием кода.
* **Ограничение видимости:** Агентам запрещено сканировать и анализировать проект целиком. Чтение и редактирование файлов строго ограничено зоной ответственности агента в текущей фазе.
* **Git-флоу:** После завершения своей фазы каждый агент обязан сделать коммит со строгим сообщением и отправить изменения в удаленный репозиторий `https://github.com/4il228/approval-service` (в ветку, указанную в `PLAN.md`).

## Инструкции по агентам

### Agent 1: Infrastructure & Init

**Цель:** Выполнение Фазы 1 из `PLAN.md`.
**Контекст спецификации:** Системные эндпоинты `GET /health`, `GET /ready` (SPEC.md, Раздел 5), стек Python + FastAPI (SPEC.md, Раздел 8).
**Права доступа:** Чтение и запись конфигурационных файлов (`Dockerfile`, `docker-compose.yml`, `pyproject.toml` / `requirements.txt`) и `main.py`.
**Инструкция:** Инициализировать пустой backend-сервис с учетом интеграции терминала Opencode. Настроить базовый роутинг. При необходимости заложить структуры под будущие интерфейсы — запросить прототип у `google-stitch` и дождаться ответа. Выполнить `git commit` и пуш в `main`.

### Agent 2: Database Architect

**Цель:** Выполнение Фазы 2 из `PLAN.md`.
**Контекст спецификации:** Работа с PostgreSQL/SQLite, обязательные миграции (SPEC.md, Раздел 8), строгая изоляция workspace (SPEC.md, Раздел 6).
**Права доступа:** Чтение и запись в директориях `models/`, `migrations/` и файле `alembic.ini`.
**Инструкция:** Разработать схему БД для хранения данных, требуемых для терминала Opencode. Убедиться, что типы полей соответствуют контрактам от `google-stitch`. Сгенерировать и применить первую миграцию. Выполнить коммит в ветку `feature/database-schema`.

### Agent 3: Security Engineer

**Цель:** Выполнение Фазы 3 из `PLAN.md`.
**Контекст спецификации:** Авторизационная заглушка и доступ по ролям (SPEC.md, Раздел 7), защита от утечек токенов/секретов в логи (SPEC.md, Раздел 6).
**Права доступа:** Чтение и запись в `middlewares/`, `core/security.py` и `main.py`.
**Инструкция:** Разработать middleware для чтения данных авторизации. Настроить валидаторы прав (например, `approval:create`) и жесткое маскирование секретов. Выполнить коммит в ветку `feature/auth-stub`.

### Agent 4: Backend API Developer

**Цель:** Выполнение Фазы 4 из `PLAN.md`.
**Контекст спецификации:** Сценарии создания и чтения (SPEC.md, Разделы 3 и 5), структура Payload (SPEC.md, Раздел 4.1), идемпотентность (SPEC.md, Раздел 6).
**Права доступа:** Чтение и запись в `api/`, `schemas/` и `services/` (только CRUD-операции).
**Инструкция:** Написать эндпоинты для создания и чтения сущностей. Дождаться подтвержденного ответа от `google-stitch` перед фиксацией JSON-схем (`Pydantic`), чтобы обеспечить полную совместимость с UI терминала Opencode. Выполнить коммит в ветку `feature/crud-requests`.

### Agent 5: Business Logic Engineer

**Цель:** Выполнение Фазы 5 из `PLAN.md`.
**Контекст спецификации:** Принятие решений Approve/Reject/Cancel (SPEC.md, Раздел 4.2), ограничение смены финальных статусов (SPEC.md, Раздел 6).
**Права доступа:** Чтение и запись в `api/`, `services/` (стейт-машина) и `models/`.
**Инструкция:** Написать логику переходов состояний. Формат полей (причины отказов, комментарии) строго синхронизировать с прототипами интерфейса из `google-stitch`. Обеспечить запись всех действий в таблицу аудита. Выполнить коммит в ветку `feature/state-machine`.

### Agent 6: QA Automation

**Цель:** Выполнение Фазы 6 из `PLAN.md`.
**Контекст спецификации:** Обязательное автоматизированное тестирование бизнес-логики (SPEC.md, Раздел 8).
**Права доступа:** Полный доступ к директории `tests/`, чтение `api/` и `services/`.
**Инструкция:** Написать интеграционные тесты на `pytest`. Проверить изоляцию тенантов, идемпотентность и переходы машины состояний. Удостовериться, что возвращаемые данные соответствуют контрактам `google-stitch`. Выполнить коммит в ветку `feature/testing`.

### Agent 7: Tech Writer & Release

**Цель:** Выполнение Фазы 7 из `PLAN.md`.
**Контекст спецификации:** Подготовка состава поставки (SPEC.md, Раздел 9).
**Права доступа:** Чтение и запись корневых Markdown-файлов (`README.md`, `DESIGN.md`).
**Инструкция:** Сформировать финальную документацию. Описать в `DESIGN.md`, как реализованный API мапится на UI-прототипы `google-stitch` и как архитектура решает задачу новой кнопки для терминала Opencode. Провести ревью, выполнить финальный мердж в `main`, установить тег `v1.0.0` и отправить в удаленный репозиторий.
