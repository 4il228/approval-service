# DESIGN.md

## Архитектурные решения approval-service

---

### 1. Модель данных

#### Таблица `approval_requests`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID (PK) | Уникальный идентификатор заявки |
| `workspace_id` | String(255) | Идентификатор рабочего пространства (мульти-тенантность) |
| `source_type` | String(50) | Тип источника: `publication`, `scenario`, `edit`, `external` |
| `source_id` | String(255) | Внешний идентификатор контента |
| `title` | String(500) | Название заявки |
| `description` | Text (nullable) | Описание заявки |
| `reviewers` | JSON | Массив ID пользователей-рецензентов |
| `status` | String(20) | Статус: `PENDING`, `APPROVED`, `REJECTED`, `CANCELED` |
| `idempotency_key` | String(255), UNIQUE, nullable | Ключ для идемпотентности |
| `created_at` | DateTime(tz) | Время создания |
| `updated_at` | DateTime(tz) | Время последнего обновления |

**Индексы:**
- `ix_approval_requests_workspace_id` — ускорение фильтрации по workspace
- `ix_approval_requests_workspace_status` — составной индекс для запросов по workspace + статус

#### Таблица `audit_logs`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID (PK) | Уникальный идентификатор записи |
| `request_id` | UUID (FK) | Ссылка на `approval_requests.id` (CASCADE delete) |
| `actor_user_id` | String(255) | ID пользователя, выполнившего действие |
| `action` | String(50) | Тип действия: `CREATED`, `APPROVED`, `REJECTED`, `CANCELED` |
| `payload` | JSON (nullable) | Дополнительные данные действия |
| `created_at` | DateTime(tz) | Время записи |

---

### 2. Границы сервиса

**Сервис отвечает только за:**
- Хранение и управление жизненным циклом заявок на согласование
- Фиксацию действий пользователей в аудит-логе
- Валидацию прав доступа и изоляцию рабочих пространств

**Сервис НЕ отвечает за:**
- Управление пользователями (ID пользователей передаются как внешние)
- Управление контентом (source_id — внешний идентификатор)
- Реальную авторизацию (используется заглушка `X-Auth-Mock`)
- Отправку уведомлений
- Фронтенд или UI

Все внешние сущности (пользователи, контент, медиафайлы) передаются только как идентификаторы.

---

### 3. Идемпотентность

Повторный POST-запрос с одним и тем же `Idempotency-Key` возвращает ранее созданную заявку (200 OK) вместо создания дубля.

**Реализация:**
1. Клиент передает заголовок `Idempotency-Key` при создании заявки
2. Сервис проверяет наличие ключа в таблице `approval_requests.idempotency_key`
3. Если ключ существует — возвращается существующая заявка
4. Если ключа нет — создается новая запись с указанным ключом

**Ограничения:**
- Ключ должен быть уникальным в пределах всей системы (UNIQUE constraint)
- Ключ не имеет срока действия — дубликация предотвращается навсегда
- Ключ применяется только к эндпоинту создания заявок

---

### 4. Шаблон Outbox (аудит-лог как основа для событий)

Таблица `audit_logs` служит основой для будущей интеграции с другими сервисами через события (Outbox Pattern).

**Принцип работы:**
1. Каждое изменение заявки (создание, approve, reject, cancel) фиксируется в `audit_logs` в рамках одной транзакции с изменением `approval_requests`
2. Внешние сервисы могут опрашивать `audit_logs` для получения событий о изменениях
3. Структура `payload` в `audit_logs` содержит все необходимые данные для восстановления контекста события

**Формат событий:**

| action | payload |
|--------|---------|
| `CREATED` | `{ source_type, source_id, title, description, reviewers }` |
| `APPROVED` | `{ comment }` |
| `REJECTED` | `{ reason }` |
| `CANCELED` | `{ reason }` |

Все записи содержат `actor_user_id` для идентификации инициатора действия.

### 5. Машина состояний

```
              ┌──────────┐
              │ PENDING  │
              └────┬─────┘
                   │
        ┌──────────┼──────────┐
        ▼          ▼          ▼
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ APPROVED │ │ REJECTED │ │ CANCELED │
  └──────────┘ └──────────┘ └──────────┘
```

**Правила:**
- Из состояния `PENDING` допустим переход в любое из трех финальных состояний
- Из финальных состояний (`APPROVED`, `REJECTED`, `CANCELED`) переходы запрещены
- Попытка изменения статуса уже завершенной заявки возвращает ошибку 400

---

### 6. Мульти-тенантность

Данные изолируются по `workspace_id`:

1. **На уровне middleware:** `validate_workspace_access()` проверяет, что `workspace_id` из URL совпадает с `workspace_id` из токена авторизации
2. **На уровне запросов:** Все SELECT-запросы фильтруются по `workspace_id`
3. **На уровне индексов:** Составной индекс `(workspace_id, status)` оптимизирует типовые запросы

---

### 7. Защита данных в логах

Фильтр `SensitiveDataFilter` перехватывает все log-сообщения и маскирует значения после ключевых слов:
- `password=`, `secret=`, `token=`, `key=`, `auth=`, `X-Auth-Mock=`

Маскированные значения заменяются на `[REDACTED]`. Фильтр устанавливается на корневой логгер при старте приложения.

---

### 8. Связь с UI (google-stitch)

На момент написания проекта прототипы интерфейсов для approval-service в google-stitch отсутствуют. Контракты API (эндпоинты, структура JSON, статусы) разработаны на основе SPEC.md и не привязаны к конкретным UI-прототипам.

При появлении прототипов в google-stitch необходимо:
1. Сопоставить эндпоинты и поля моделей с экранами прототипа
2. Обновить Pydantic-схемы при расхождении структур данных
3. При необходимости добавить/изменить эндпоинты для соответствия UI

### 9. Известные компромиссы

| Компромисс | Описание | Потенциальное улучшение |
|------------|----------|------------------------|
| Auth-заглушка | Реальная авторизация не реализована; используется заголовок `X-Auth-Mock` | Интеграция с OAuth/OIDC провайдером |
| Нет распределенных транзакций | Идемпотентность и аудит работают в рамках одной БД | При масштабировании — переход на Outbox + Kafka/RabbitMQ |
| Stateless state machine | Статус хранится в `status` поле; нет явного описания допустимых переходов | Выделенный класс `StateMachine` с декларативными правилами |
| Нет пагинации | `GET .../approval-requests` возвращает все заявки workspace | Добавить `limit`/`offset` или курсорную пагинацию |
| Нет soft delete | Заявки физически удаляются при каскадном удалении | Добавить поле `deleted_at` для мягкого удаления |
| Ключ идемпотентности навсегда | Нет TTL для ключей; уникальность гарантирована навсегда | Добавить механизм очистки старых ключей |
| SQLite в тестах | Тесты используют SQLite, продакшн — PostgreSQL | Некоторые PG-специфичные фичи не тестируются |
